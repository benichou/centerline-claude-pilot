## 2. Must-fix

### B3 — Recurring timeout on the Arcadia cross-source reconciliation prompt

**Finding:** In the latest run (2026-06-17), B3 produced two `ERROR — timeout` results and one pass. The same prompt produced 1/3 on 2026-06-14 03:55. All other prompts pass clean in those same runs, ruling out infrastructure. The B3 prompt is the Arcadia cross-source reconciliation task — the most complex prompt in the eval set.

**Root cause:** `detecting-cross-source-discrepancies/SKILL.md` Step 1 currently prescribes an unconditional pull of five MCP tools:

> *"…Also pull `get_emails(borrower)` and `get_activity_log(borrower)` for the full text, and `get_loan_performance` / `detect_deterioration_signals` for the structured figures."*

Two problems with this wording:

1. **"Also pull … for the full text"** is read as an unconditional instruction. When the model calls `get_relationship_timeline` (which already merges the CRM log and email messages into a tagged, dated list) and then *also* calls `get_emails` and `get_activity_log` separately, it is loading the same underlying content twice, inflating context and execution time.

2. **"`get_loan_performance` / `detect_deterioration_signals`"** — the slash is ambiguous and the model sometimes calls *both* rather than choosing one. `detect_deterioration_signals` already returns the loan performance figures needed for figure-contradiction checks; calling `get_loan_performance` separately on top of it is redundant.

The one passing run (run 3, 2026-06-17) used exactly: `detect_deterioration_signals`, `get_activity_log`, `get_emails`, `get_relationship_timeline`, `screen_and_finalize`, `render_pdf` — six calls. The timeout runs presumably called more, or called them in a way that created a deeper reasoning loop before producing output.

**Proposed skill/prompt change — `detecting-cross-source-discrepancies/SKILL.md`, Step 1:**

Replace the current Step 1 (lines beginning "Pull the merged timeline…") with:

> **1. Pull the merged timeline.**  Call the `centerline` MCP tool **`get_relationship_timeline(borrower)`** — it merges the CRM log + the email messages into one chronological, source-tagged list (deterministic dates, so you don't reconstruct them). **Start here; this is the primary source.**
>
> Pull `get_emails(borrower)` and `get_activity_log(borrower)` **only if** you need full verbatim text that the merged timeline summary does not include — for example, to quote a credit-officer decision word-for-word. Do not pull both unless you need both; do not pull either if the timeline entry already contains the phrase you need to quote.
>
> For figure or date contradictions involving structured data, call **`detect_deterioration_signals`** — it covers loan performance figures. Do **not** also call `get_loan_performance` separately unless `detect_deterioration_signals` does not return the specific data point in question.
>
> The goal is to surface the three discrepancy types below — pull only the sources you actually need to do that.

**Compliance rationale (§4.2, §4.3):** This change affects only tool-call sequencing and scope, not what is done with the results. The skill still requires routing every output through `screen_and_finalize` (§4.2 guard), and still requires the `render_pdf` + RM-review tag (§4.3). No compliance logic is touched.

**Why this does not weaken any guard or test:** The eval's tool-selection check for B3's passing run already expects `detect_deterioration_signals`, `get_activity_log`, `get_emails`, `get_relationship_timeline`, `screen_and_finalize`, `render_pdf` — the proposed wording makes those calls conditional on need, but they remain the expected set for a prompt that requires verbatim email quoting and figure contradiction checks (as the Arcadia B3 prompt does). The MCP guard in `guards.py` is not referenced and is not proposed for change. The eval answer key case is not referenced and is not proposed for change.

**Severity:** Low — absent in the last three run sets; likely a use-vs-mention eval artifact in the prior runs. The skills already contain the correct guard. Including this only because there are two separate run dates showing it.

**Observation:** Both `assembling-watchlist-triage/SKILL.md` and `measuring-engagement-coverage/SKILL.md` already instruct the model not to write negation-disclaimers ("naming credit vocab even in negation trips the §4.2 scanner"). Yet on 2026-06-14 03:55 (A1 and A3) and 2026-06-14 17:13 (A3), a §4.2 hit appeared on one run each. The `screen_and_finalize` call was not itself the problem — the model appears to have written a prose disclaimer before or alongside calling the tool.

**Why this is likely a use-vs-mention artifact, not a real compliance failure:** The eval's own "Honest limits" note calls out exactly this pattern: *"the scanner flags a blocked phrase even if the agent is quoting it to explain a block."* The successful runs (including all runs in the last three run sets) produce clean §4.2 output even when they include phrases like "facts-derived triage order — not a credit rating," which suggests the scanner allows some constructions and only trips on others (e.g., "not a creditworthiness assessment," "no credit risk characterization"). The model has been clean for three consecutive run sets, suggesting the existing guard is working.

**Proactive hardening (specific and actionable):** Each of the two skills has a prohibition ("do not…") but no concrete positive example of what to write when the model wants to signal the output's nature. Adding one example phrase reduces the chance the model reaches for a negation-construct when the prohibition fires.

For **`assembling-watchlist-triage/SKILL.md`**, after the existing line *"naming credit vocab even in negation trips the §4.2 scanner"*, add:

> *If you need to label the ranking methodology in the output, use: "facts-derived triage order" — that phrase is already in the approved output format and does not trigger the §4.2 screen.*

For **`measuring-engagement-coverage/SKILL.md`**, after the existing prohibition line, add:

> *To characterize the output's scope in the artifact, use: "days since last two-way contact" or "engagement gap (log-derived)" — these are factual and do not trigger the §4.2 screen.*

These additions give the model a fall-back positive phrase, reducing the probability it invents a negation-disclaimer under uncertainty.
